# 修复计划: processNodes 接口分支节点排序异常

> 编码: 116393 | 日期: 2026-05-14 | 类型: Bug 修复 | 来源: bugfix | 基于: docs/bugs/116393-并行人工包容分支节点排序错误.md

## 根因分析

问题位于 `FlowProcessNodeService.NextNodeLoader.fetchNextNode()`（`flow-engine-framework/src/main/java/com/codingapi/flow/service/impl/FlowProcessNodeService.java:174-197`），递归逻辑是**深度优先（DFS）**：

```java
this.nodeList.add(processNode);                                    // 先 add 当前
List<IFlowNode> nextNodes = workflow.nextNodes(flowNode);
List<IFlowNode> selected = selectBranchNodes(nextNodes, ...);
this.fetchNextNode(flowSession.updateSession(flowNode), selected); // 再递归
```

对并行（`ParallelNode`）/ 人工（`ManualNode`）/ 包容（`InclusiveNode`，当多分支同时满足）等"必须走多条分支"的控制节点，遍历会**先把第一条分支一路走到 End**，再回溯处理其它分支：

```
顺序: Start → ParallelNode → Branch_1 → Approval_A → ParallelEnd → End
           ↑ DFS 一路走到底
      → (回溯) Branch_2 → Approval_B   ← 此时才 add，已经在 End 之后
      → (回溯) Branch_3 → Approval_C
```

`NextNodeLoader.loadNextNode()`（`FlowProcessNodeService.java:229-240`）最后只做了 `isDisplay` 过滤和"重复 nodeId 去重"，**没有按节点定义顺序重排**，因此最终对外返回的列表顺序就是 DFS 的访问顺序：`[Start, Approval_A, ParallelEnd, End, Approval_B, Approval_C]`，分支节点错位在 End 之后。

而条件分支（`ConditionNode`）不受影响，因为 `selectBranchNodes`（`FlowProcessNodeService.java:199-227`）会把条件分支收敛到单条路径，DFS 不会回溯到其它分支。

证据：
- `FlowProcessNodeService.java:174-197` DFS 递归未做后置排序
- `FlowProcessNodeService.java:199-227` 当前 `selectBranchNodes` 只覆盖 `ConditionBranchNode` / `InclusiveBranchNode` 等 4 个类型，且对 if 命中的情况会单选一条
- `ParallelBranchNode.filterBranches`（`ParallelBranchNode.java:50-65`）返回全部分支节点（不收敛），由此走 DFS 全遍历
- `ManualNode.filterBranches`（`ManualNode.java:45-54`）在无 advice 时返回全部分支
- 节点的"定义顺序"权威来源是 `Workflow.nodes`（`Workflow.java:108、162-164`），它是按 `WorkflowBuilder.addNode` 顺序追加的有序 `List`

## 影响面评估

- 直接修改的 Maven 模块: `flow-engine-framework`
- 可能回归的场景:
  - 含并行节点（`ParallelNode`）的流程预览顺序
  - 含人工节点（`ManualNode`）的流程预览顺序

  - 已运行流程（含历史节点 + 未来节点混合返回的场景）
- 是否需要数据迁移/配置变更: 否

## 关键代码修改

`NextNodeLoader.loadNextNode()` 当前实现：

```java
public List<ProcessNode> loadNextNode(FlowSession flowSession) {
    this.fetchNextNode(flowSession, List.of(this.currentNode));
    List<ProcessNode> displayNodes = nodeList.stream().filter(ProcessNode::isDisplay).toList();
    List<ProcessNode> processNodeList = new ArrayList<>();
    for (ProcessNode node : displayNodes) {
        if (!processNodeList.contains(node)) {
            processNodeList.add(node);
        }
    }
    return processNodeList;
}
```

修改为：

```java
public List<ProcessNode> loadNextNode(FlowSession flowSession) {
    this.fetchNextNode(flowSession, List.of(this.currentNode));
    List<ProcessNode> displayNodes = nodeList.stream().filter(ProcessNode::isDisplay).toList();
    List<ProcessNode> processNodeList = new ArrayList<>();
    for (ProcessNode node : displayNodes) {
        if (!processNodeList.contains(node)) {
            processNodeList.add(node);
        }
    }
    return sortByWorkflowNodeOrder(processNodeList);
}

private List<ProcessNode> sortByWorkflowNodeOrder(List<ProcessNode> processNodes) {
    List<IFlowNode> workflowNodes = workflow.getNodes();
    Map<String, Integer> indexMap = new HashMap<>();
    for (int i = 0; i < workflowNodes.size(); i++) {
        indexMap.put(workflowNodes.get(i).getId(), i);
    }
    // 未在 workflow.nodes 顶层注册的节点（如嵌套子节点）放到末尾，相对顺序保持稳定
    return processNodes.stream()
            .sorted(Comparator.comparingInt(
                    n -> indexMap.getOrDefault(n.getNodeId(), Integer.MAX_VALUE)))
            .toList();
}
```

要点说明：

1. **以 `Workflow.nodes` 索引为权威排序键**：该列表在 `WorkflowBuilder.addNode()` 中按调用顺序追加，对应前端流程图保存时的节点顺序，是"定义顺序"的事实依据
2. **稳定排序**：`Comparator.comparingInt` 配合 `Stream.sorted` 是稳定排序，索引相同（如未注册节点）的元素保持 DFS 顺序，不会引入新的不确定性
3. **不动 `fetchNextNode` 内部逻辑**：递归 add 顺序保持原状，仅在最终输出前做一次重排。避免影响 `processNode.setCurrentState()` 等中间状态判定
4. **不影响历史节点**：`processNodes()` 在调用 `loadNextNode` 之前已经把历史节点 add 到 `this.nodeList` —— 这里返回的 `processNodeList` 只是"未来节点"部分，外层 `nodeList.addAll(nextNodes)` 仍把它们追加在历史节点之后
